.: HOW TO Revolver-DS :.

a semantical ITCSS DS methodology

I had this DX question: what would the perfect developer hand-off to any design system by a designer look like?... Well, like any other design system!

To get YOU and designers started with Revolver-DS;

  1. Fork the Repository from Github
  2. Create a new branch for their changes and additions
  3. Import their changes using a pull request in due time
  4. Add .rds class to your body (or anything small)
  5. Done! Just lock, load & ship their designs, when they're ready!

It was designed for lethal and non-destructive simplicity on language, all buff and no technical fluff other than the latest and greatest CSS. Like a gun, a CSS design system can be a scary thing to see or to handle but it's not; it's simple if you play it safe!

Revolver-DS unloaded

Revolver-DS is made with one simple strategy; NO Docs, strictly semantical principles and product development. Since there are no docs, all code is strictly commented and naming things semantically is a top priority. That leaves prototypes and demos as documentation (like this post).

Revolver-DS is a simple revolver-themed methodology in CSS; just Lock, Load & Ship your non-destructive CSS using only the @layer syntax, which is a CSS-only syntax that allows you to @import a CSS file into a specific semantical layer of a Revolver-DS.


        /*style.css*/

        @layer reset, lock, load, ship, preset;

        @import url('../css/lock.css') layer(lock);
        @import url('../css/load.css') layer(load);
        @import url('../css/ship.css') layer(ship);

Key Terminology

From the ground up, Revolver-DS' design language is made for designing design systems, not designers or developers. The entire system is 100% semantically named, has no docs, and thus needs some introduction of it's simple language use;

Four Types of Variables

  1. Literal Variables
  2. Nested Variables
  3. Relative Variables
  4. Lateral Variables

I've written about these types of variables in another post, but they are quite simple to understand without reading up on it; Literal variables hold literal (hardcoded) values, that relative variables consume as they hold literal or nested variables they relate to. Nested Variables are a special kind of literal variable, like --color--blue and --_color--blue would be linked in this example. They nest their parents name, and only serve "family" purpose.

Lateral Variables are design tokens that hold multiple values loaded elsewhere, like how a hsl(var(--h), var(--s), var(--l)) would make up a HSL-color. They are laterally connected design tokens, hence the name lateral token.

Five Types of @Layers

  1. @Reset
  2. @Lock
  3. @Load
  4. @Ship
  5. @Preset

Each kind of token also has a special layer that it belongs to, and each layer has one writing rule attached to write tokens for the three primary layers. This is done for the sake of layering and the ease of remembering "which token goes where" in the loading order of Revolver-DS.

@layer Reset

This is the traditional place to put your CSS reset. Revolver-DS has one included but you are free to replace it with your own, or even unload it completely by locking out the @reset layer.


    /* Revolver DS Design-Reset, based on Eric Meijer's, Elad Shechter, Andy Bell 
    & Josh Comeau's CSS resets for sane defaults in CSS */

    body, iframe, pre, img, svg, video, canvas, select {
            max-width: 100%;
            overflow: auto;
            word-break: break-word;
        }

    *:not(iframe, canvas, img, svg, video):not(svg *)) {
        all: unset;
        display: revert;
    }
    /* and some more .... */

@layer Lock

LOCK.css is specifically used to lock in all your hard-coded values. These are referred to as "Literal Design Tokens" as they hold literal values. It is recommended that you annotate every literal token with an _underscore before the prefix and after --. This is so you allways know which css file and which layer the token comes from in the order of loading or when you are debugging.


        /* color secundairy, abbreviated to first-letters */

        --_cs__mob: ;
        --_cs__ooob: ;
        --_cs__oob: ;
        --_cs__ob: ;
        --_cs: #face22;
        --_cs__ub: ;
        --_cs__uub: ;
        --_cs__uuub: ;
        --_cs__mub: ;

If you are unfamiliar with BOUM-tokens , they are a dead-simple way to name tokens on a ramp semantically from a relative base. It stands for Base, On, Under, Max | Modifier and is a companion to naming things the BEM-way semantically & simple.

There are three bonusses for using BOUM-tokens for developers. It doesn't conflict with any excisting naming convention, it can flip colors from one side of the scale to the other by just replacing o's and u's programmatically, and it comes with a free custom open source copyright license agreement, designed specifically for handling DS artifacts internally & externally.

Sorry, No NFT's allowed

If you would like to work with Revolver-DS without BOUM-tokens because of the license agreement, just replace the tokens with 100~900 variations you see in most design systems to date. @layer Load

LOAD.css is used to load in all your literal and nested design tokens. These are referred to as "Relative Design Tokens" as they are relative to other tokens. This is where you can load in your own design tokens, or even unload them completely by locking out the @load layer.

If you followed the rule of _underscoring in lock.css, in load.css you can now use the same name minus the _underscore. This makes sure that you know exactly which layer or css-file the token is coming from; lock or load.


        /* color secundairy, abbreviated to first-letters */

        --cs__mob: var(--_cs__mob);
        --cs__ooob: var(--_cs__ooob);
        --cs__oob: var(--_cs__oob);
        --cs__ob: var(--_cs__ob);
        --cs: var(--_cs);
        --cs__ub: var(--_cs__ub);
        --cs__uub: var(--_cs__uub);
        --cs__uuub: var(--_cs__uuub);
        --cs__mub: var(--_cs__mub);

@layer Ship

SHIP.css is specifically used to lock in all your Lateral Tokens (loaded elsewhere), and are instantly recognizable by their 'u' prefix that means utility, followed by their group identity like color(--uc), layout(--ul), font(--uf).

This is the whole reason to stick to the naming convention, so it doesn't need documentation to see where something is coming from, and can be explained in a comment.


        /* RELATIVE utility colors: LITERAL selector */
        --uc__background: var(--cs);
        --uc__foreground: var(--cb);
    
        --uc__text--mob: var(--);
        --uc__text: var(--cb);
        --uc__text--mub: var(--);
    
        --uc__fill: var(--uc__foreground);
        --uc__border: var(--uc__foreground);
        
        /*  utility layout: Spacing */
        --ul__space--ob2: clamp(3.75rem, calc(2.00rem + 8.75vw), 12.50rem);
        --ul__space--ob1: clamp(1.88rem, calc(1.00rem + 4.38vw), 6.25rem);
        --ul__space--base: clamp(1.0rem, calc(0.50rem + 2.19vw), 3.13rem);
        --ul__space--ub1: clamp(0.50rem, calc(0.29rem + 1.06vw), 1.56rem);
        --ul__space--ub2: clamp(0.13rem, calc(0.08rem + 0.25vw), 0.38rem);

@layer Preset

Preset is where your design tokens and your (dev) design system meet and sync up their values. In or outwards, you could set up Revolver-DS to consume your tokens, or in reverse. Non-destructively.


        /* from Revolver-Ds to your design system */
        --cs__mob: var(--color-primary--950);
        --cs__ooob: var(--color-primary--900);
        --cs__oob: var(--color-primary--800);
        --cs__ob: var(--color-primary--700);
        --cs: var(--color-primary--600);
        --cs__ub: var(--color-primary--500);
        --cs__uub: var(--color-primary--400);
        --cs__uuub: var(--color-primary--300);
        --cs__mub: var(--color-primary--200);

        /* from your design system to Revolver-Ds */
        --color-primary--950: var(--cs__mob);
        --color-primary--900: var(--cs__ooob);
        --color-primary--800: var(--cs__oob);
        --color-primary--700: var(--cs__ob);
        --color-primary--600: var(--cs);
        --color-primary--500: var(--cs__ub);
        --color-primary--400: var(--cs__uub);
        --color-primary--300: var(--cs__uuub);
        --color-primary--200: var(--cs__mub);

This makes sure that you are never stuck having to adopt to Revolver-DS, or the other way around if you're a designer wanting to consume the front-end design system without leaving a trace of bad code. A headless design system for people who don't like to lose their head in the maze!

Nesting RDS @layers

Nested design tokens carry the same name as their parent. Nested @layers do too and each of the layers Lock, Load and Ship can be subdevided with the same set;


        @layer lock.lock {}
        @layer lock.load {}
        @layer lock.ship {}
        @layer load.lock {}
        @layer load.load {}
        @layer load.ship {}
        @layer ship.lock {}
        @layer ship.load {}
        @layer ship.ship {}

This is done so that once you have a loading strategy that you like and want to stick to at scale,it can easily be maintained on the spot in case you need to import something inbetween two layers, like a bunch of utilities for example or maybe even a testing library;


        @import url('lock.css') layer(lock.lock)
        @import url('color--utilities.css') layer(lock.load) 
        @import url('load.css') layer(load)
        @import url('ship.css') layer(ship.lock)
        @import url('tokens.css') layer(ship.load)
        @import url('testkit.css') layer(ship.ship)

Making sure everything is nicely put into semantical layers keeps the system fresh without the tech jargon and naming! It can concurrently recreate or subdevide itself with itself, which is an axiomatic holy grail in abstract factory modelling. In this case it's done by one designer and the new CSS ;)

@layer *

In your style.css file there is one last layer; the anonymous @layer without a name. This works like any other layer except you can't add anything to that layer later on (by intent), or it will conflict with itself. This is the last layer you will need to use before any regular CSS kicks in.


        @layer {
            p, h1, h2, h3 {
                max-width: clamp(46ch, 50vw, 68ch);
                margin-inline-start: clamp(2vw, 2.5vw, 3vw);
            }
            
            h1, h2, h3 {
                margin-block-end: clamp(-0.35vw, -0.6vw, -1vw);
                white-space: nowrap;
            }
            
            p, h2, h3, ul, ol {
                font-family: "IBM Plex Mono", Arial, monospace;
            }

The MOST !important @layer

@layers support !important in a very special way; they reverse the order of loading @layers! And Revolver-DS does NOT USE !important in any of it's layers. Meaning?...

This means you still have HALF of all the @layers available that Revolver-DS could provide for you technically speaking! This "negative space" is intentionally left for your design of system to exploit Revolver-DS' loading order JIT, without JS!

For a good read on what !important can do already using @layers, have a look what Miriam Suzanne did with it, one of the (if not the only) authors of the new @layer feature in CSS, and creater of Sass
